Linux常用操作指令总结

Posted on 2019-10-15

网上相关的文章有很多，但是比较松散，这里会作为一个整体上的总结，希望能加深自己的记忆（以 Debian 系统为例）。

信息查询

这一类型指令包含查询当前 Linux 系统的一些基本信息，例如内存、CPU、磁盘等等，有助于对服务器的原始状态有一个初步了解。一些基本指令是基于 dmidecode 工具的，如果未找到指令，使用 sudo apt-get install dmidecode 安装。

服务器

  1. 查看服务器品牌型号信息

    dmidecode | grep -A2 "System Information" | grep "Product Name"

内存

  1. 查看每个插槽的内存大小

    dmidecode | grep -A5 "Memory Device" | grep Size

  2. 查看每个插槽的内存频率

    dmidecode | grep -A16 "Memory Device" | grep "Speed"

CPU

  1. 查看 CPU 个数、核心数

    • 物理 CPU 个数：cat /proc/cpuinfo | grep "physical id" | sort | uniq | wc -l
    • 逻辑 CPU 个数：cat /proc/cpuinfo| grep "processor"| wc -l
    • 查看单个 CPU 核心数：cat /proc/cpuinfo| grep "cpu cores"| uniq

    另外要关注的是如下两个关于 CPU 的计算公式

    • 总核心数 = 物理 CPU 个数 * 单个 CPU 核心数
    • 逻辑 CPU 个数 = 物理 CPU 个数 单个 CPU 核心数 超线程数

  2. 查看 CPU 型号

    cat /proc/cpuinfo | grep name | cut -f2 -d: | uniq -c

    PS: 可以留意下这里 cut 分割字符串的用法，比 awk 简单直接一些。

  3. 查看 CPU 频率

    cat /proc/cpuinfo | grep "cpu MHz"

磁盘

查看磁盘的场景应该是最多的，例如我们需要确认磁盘状态、磁盘类型，从而进行一些维护处理，都需要用到磁盘查看的相关指令。

  1. 确认磁盘是否为 SSD

    • 利用 block layer 特性判断：我们以 sda 为例，通过 cat /sys/block/sda/queue/rotational 指令查看返回，如果为 0 则为 SSD 盘，返回 1 则为普通 HDD。然而，这个方式也有不准确的可能性，具体可以参考 此链接
    • 利用 lsblk 指令判断：其实跟前面的方法类似，使用 lsblk -d -o name,rota 指令用于一次性显示各个磁盘是否为 rotational。
    • 使用 smartctl 指令判断：此指令包含在 smartmontools 包中，使用 smartctl -i /dev/sda 查看磁盘信息（做了 raid 的盘目前也不是很清楚怎么进行查询判断）。另外关于 SMART 工具，可以参考 此文章

    所以，上述方法最准确的使用场景应该是在没有做 Raid 的情况下。如果在做了 Raid 的情况下，我们可以使用以下方法。

    • 利用 hwtool 指令判断：使用 hwtool -m raid -d 指令的输出，即可查看磁盘信息，以及最重要的 磁盘状态 也可以使用此命令查询到
    • 利用 MegaCli 指令判断：使用 /usr/local/hwtool/tool/MegaCli PDList aAll 指令的输出，即可查看物理盘的具体信息，同样，磁盘状态 也可以使用此命令查询到

使用 eval 函数

在我上面提到的任务中，eval() 函数可以极大的简化数据处理流程。

但是值得注意的是，这个函数同时也 非常的危险（很多书籍中都有提到），因为一旦使用不当，被人利用，会有不可预计的后果。例如下面这段输入，就会被用来移除系统中的文件。

用Python去解释何为倒排索引

Posted on 2019-06-03 | Edited on 2019-07-09

首先，本文基于参考此博文。

因为最近想到既然自己在做 ELK 相关的内容，那么就要了解下 ES 的本质，加上自己最熟悉的语言就是 Python，所以就顺手上网搜了下前人总结的一些相关经验。本文也是对这些内容的二次消化，加上一些自己对于某些细节部分的解读。当然本文的理解可能也只停留在一个浅显的层面，没有深入的研究源码，但是这也是为之后的学习开一个头。

理解倒排

倒排索引，其实是定义的一种查询方式，直接通过中文理解很容易产生歧义。但是看到其英文 – Inverted Index，就能理解这个 倒 究竟是什么意思了。

这里的倒，并不是我们的数据在储存顺序上的颠倒，更贴切的说，是一种查询逻辑上的颠倒。

通俗来讲，我们原有如果是通过 *索引/ID** A 找到其对应的数据 B，那么就可以说这是一种 正排索引 逻辑；反之，如果是通过数据 B，找到其 *索引/ID** A (当然，还有其他一些信息)，那么就可以说是一种 倒排索引 逻辑。

那么倒排索引，为什么能够被广泛的应用在全文检索的场景中，从上面的理解中，答案也就顺理成章的浮现出来了。

如果想想一下用 正排索引 的逻辑，去做一个文本的搜索，我们要怎么查？因为我们不知道数据 B 中有没有要查的文本，只能每条记录逐一检查，然后返回符合条件记录的相关信息，效率想一想就很低。

如果遵循 倒排索引 的逻辑，那么我们相当于知道了哪些 *索引/ID** 对应的记录包含了需要搜索的文本内容，从感觉上 (是的，暂时是从感觉上)，效率就比较高。当然这种高效，是在全文检索的这一个使用场景下，因为避开使用场景的讨论，都是不实际的。

PS: 这里稍微回头想一下，为什么 ES 中做分词、不做分词、用不同的分词器分词，搜索出来的文档结果集不一样呢？那是因为前面所说的各种情况，所生成的倒排索引不同，那么所得到的 B to A 的结果就各不相同了。

扁平化与反扁平化字典

扁平化字典的需求，其实是在做开发的时候遇到的问题，就是希望把一个嵌套类型的字典，如 {'a': {'b': {'c': 1}}}，转换成为如{'a.b.c': 1}这种形式，其根本目的是希望能够让前端的程序，在列表中，更好的展现这个字典的内容。

代码整体的思路比较简单，首先设定一个限于函数内存的栈，用来缓存处理的信息，而推入栈中的内容，是一个元组，包含以下两个项：

  1. 当前的字段列表，即能够表明，这个元组表示的是字典中的哪一个节点
  2. 该节点的内容，即表示，在这一层中，数据的内容是什么

那么捋一捋之后就比较清楚整体的逻辑了，即从整个字典开始扫，如果发现下一层元素的值仍然是个字典，那么，将这个元素的 key 以及 value 放到 stack 中，等待 while 循环进行处理。
